The Easiest Way to Have a Blog

I started writing in 2016, registering a 10-year domain to host technical blogs on machine learning: backpropagation formula derivations, RNN/LSTM principles. Each post was a labor of love. Using Jekyll with a sleek theme, I fussed over fonts, equations, and layout. Three years later, I’d forgotten the blog’s URL.

From 2018 to 2022, I didn’t blog at all. The thought of domains, fonts, configs, and deployment was overwhelming. Then one day, I wanted to explain Stable Diffusion’s principles under a platform-independent identity. This time, I skipped domain registration—hence the current mazzzystar.github.io—aiming for extreme simplicity.

I chose Hexo, with its complex setup. The workflow: hexo new 'title', hexo g to render, preview on localhost:4000, hexo d to publish, wait a couple minutes, see it on Github Pages.

Hexo is decent, but not perfect. Writing in local VS Code, switching to browser for preview, manually uploading and linking images—it all felt clunky.

Later, I wanted to add a ‘thoughts’ tab for short, public musings—free from social media’s distorting metrics. But Hexo made this difficult: blogs are article lists, thoughts are text cards. It took two days to code this feature.

Realizing friends had similar needs, I wondered: could there be a truly simple, foolproof platform for rapid blog and thought publishing?

The challenge wasn’t technical, but trust. Users worry about data security and potential loss if a platform shuts down. I considered various solutions: self-hosted databases, daily auto-exports. All too cumbersome and impractical.

Then it hit me: Github itself could do the job. For developers, it’s a near-immortal platform that generously stores user data. What if we kept each user’s blog and thoughts in their Github account?

Thus, Tinymind was born. After Github authorization, it creates a ‘tinymind-blog’ repo in your account. Each blog post or thought becomes a commit. It’s serverless and open-source.

I launched this website last week and completed all features yesterday. In under a week, over 400 people, many without coding experience, created public blogs visible in their Github ‘tinymind-blog’ repos. With Markdown support, real-time preview, and drag-and-drop image uploads, it’s far more convenient than my current setup.

Am I migrating? No, but I’ll use it to draft and preview before finalizing on mazzzystar.github.io. I’ll jot down raw ideas in Tinymind, then refine them for my Thoughts page. When things become simple, people do them more often.

Why must the content be public? Because I don’t want Tinymind to request access to your private repositories, which means it can only create publicly visible repositories. At the same time, your blog data is stored in your own Github account, not hosted on Tinymind.

I’ve spent late nights refining Tinymind to my standards. Now it’s complete, and I’m moving on to the next project. But I hope it enables many non-programmers to have personal control over their data and blogs.
